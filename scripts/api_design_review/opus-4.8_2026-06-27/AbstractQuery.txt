1. OVERALL ASSESSMENT
AbstractQuery<Stmt,This> is the fluent base for PreparedQuery/NamedQuery/CallableQuery; the bulk of
it is well-structured and consistent. The setXxx(int, primitive|wrapper|wrapper+default) family is
uniform (primitive, null-handling wrapper, and @Beta default-for-null triplets) with correct JDBC
Types mapping (including the deliberate Float->REAL fix). The terminal-op families are internally
consistent: queryForBoolean/Char/Byte/Short/Int/Long/Float/Double return OptionalXxx;
queryForString/BigInteger/BigDecimal/Date/Time/Timestamp/Bytes return Nullable;
queryForSingleValue/UniqueValue (Nullable) pair cleanly with queryForSingleNonNull/UniqueNonNull
(Optional); findFirst/findOnlyOne (Optional) mirror findFirstOrNull/findOnlyOneOrNull (null); and the
Class / RowMapper / BiRowMapper triad is offered consistently across findFirst, findOnlyOne, list,
stream, insert, batchInsert. The findings concentrate on one genuine duplicate, several
deprecation-consistency gaps, and a few capitalization-based naming hazards.

RE-VERIFICATION SUMMARY: Of the 13 findings, 11 are STILL-VALID, 1 is PARTIALLY-ADDRESSED
(settParameters now has a class-doc disambiguation block, but is not renamed), and 1 was already
correct and remains so (NO-LONGER-APPLIES — no action). The known package renames (JdbcUtils ->
DataTransferUtil, ColumnGetter apply->get / get->forType, RowExtractor.createBy->forType, etc.) do
NOT touch AbstractQuery's public surface: this file references none of those renamed symbols (grep
for createBy/createForArray/ColumnGetter/JdbcUtils/getColumnNameList = 0 hits), and its `.apply(...)`
call sites are all RowMapper/BiRowMapper/ResultExtractor (SAM still `apply`). No NEW public-API
inconsistencies were introduced.

2. NAMING ISSUES
- [STILL-VALID] foreach(Consumer<DisposableObjArray>) / foreach(Class, Consumer<DisposableObjArray>)
  vs the forEach(RowConsumer ...) family | AbstractQuery.java:8941, 8995 (vs forEach 8696/8738/8790/8848) |
  Medium | The two families differ ONLY by letter case (foreach vs forEach) and take different
  callback contracts; a one-character typo silently selects a different method. The Javadoc documents
  the hazard (8908-8911) and an in-source "NAMING-REVIEW SUGGESTION (2026-06-27)" comment at
  8937-8939 / 8991-8993 proposes rename `forEachRow`, but no rename was made.
  Current evidence: `public void foreach(final Consumer<DisposableObjArray> rowConsumer) throws SQLException { //NOSONAR` (8941);
  `public void foreach(final Class<?> entityClass, final Consumer<DisposableObjArray> rowConsumer) throws SQLException { //NOSONAR` (8995).
  Recommend: add an unambiguous name (e.g. forEachRow) and keep foreach only as a deprecated alias.
- [PARTIALLY-ADDRESSED] settParameters(...) (double-t) vs setParameters(...) | AbstractQuery.java:3152, 3199 |
  Low | Same capitalization/typo hazard; the double-t disambiguates ParametersSetter<? super This>
  from ParametersSetter<? super Stmt>. SINCE THE STALE REVIEW, a class-level doc block was added
  (lines 115-125) explicitly explaining the distinction ("the extra {@code t} is not a typo, and
  because both compile, the compiler will not flag a wrong choice"), which addresses the
  documentation gap; the naming hazard itself remains (no rename).
  Current evidence: `public This settParameters(final Jdbc.ParametersSetter<? super This> parametersSetter)...` (3152);
  class-doc `<li>{@code settParameters(...)} instead hands you <i>this</i> query ({@code This})...` (122).
  Recommend (optional): a clearer name (configParameters / setParametersForQuery). Confirming check
  noted in stale report stands: a single setParameters name would cause lambda-inference ambiguity.

3. RETURN-TYPE AND PARAMETER ISSUES
- [STILL-VALID] query2ResultSets / query3ResultSets accept only BiResultExtractor |
  AbstractQuery.java:5505, 5571 | Low | The single-result query(...) and queryAllResultSets(...) each
  offer BOTH ResultExtractor and BiResultExtractor variants (queryAllResultSets at 5662 ResultExtractor /
  5718 BiResultExtractor), but the numbered multi-result-set methods offer only BiResultExtractor.
  (CallableQuery's query2/query3ResultSetsAndGetOutParameters share the limitation — cross-class.)
  Current evidence: `public <R1, R2> Tuple2<R1, R2> query2ResultSets(final Jdbc.BiResultExtractor<? extends R1> resultExtractor1, final Jdbc.BiResultExtractor<? extends R2> resultExtractor2)...` (5505-5506).
  Recommend: add the ResultExtractor overloads for parity, or document the intentional restriction.
- [STILL-VALID — intentional omission documented in-source] setParameters(String,String) exists but
  no setParameters(Object,Object) | AbstractQuery.java:2270 (fixed-arity String overloads start at 2
  args) vs 2452 (Object overloads start at 3; the 2-Object form is commented out at 2419-2434) | Low |
  A caller with two Object params cannot use setParameters(a,b) though the String case works. The
  confirming check is ANSWERED by the source: the 2-Object overload is commented out with the note
  "@deprecated to void error: java ambiguous method call with other setParameters methods" (2426) —
  i.e. adding it WOULD create overload-resolution ambiguity. So the omission is deliberate.
  Current evidence: `public This setParameters(final String param1, final String param2) throws SQLException {` (2270);
  `//    public This setParameters(final Object param1, final Object param2) throws SQLException {` (2429, commented).
  Recommend: leave as-is and (optionally) add a one-line note on the String overload pointing callers
  to the 3+-arg Object forms or setParameters(Collection)/setParameters(T, BiParametersSetter) (2806).

4. MISPLACED METHODS
- [STILL-VALID] None. Every public method belongs on this fluent base; subclass-specific concerns
  (named params, out-params) are correctly absent.

6. DEPRECATION / REMOVAL CANDIDATES (with replacement APIs)
- [STILL-VALID] setBigIntegerAsString(int, BigInteger) duplicates setString(int, BigInteger) |
  AbstractQuery.java:1082 vs 1182 | High | VERIFIED byte-for-byte equivalent — setBigIntegerAsString's
  body is literally `return setString(parameterIndex, value);` (1083); each @see's the other; both @Beta
  (1081 / 1181). setString(int, BigInteger) binds the BigInteger as a base-10 VARCHAR (1186
  `stmt.setString(parameterIndex, value.toString(10));`).
  Current evidence: `public This setBigIntegerAsString(final int parameterIndex, final BigInteger value) throws SQLException { return setString(parameterIndex, value); }` (1082-1084);
  `@Beta public This setString(final int parameterIndex, final BigInteger value)...` (1181-1182).
  Recommend: keep the explicitly-named setBigIntegerAsString and remove the surprising
  setString(int, BigInteger) overload (a setString taking BigInteger is itself misleading).
- [STILL-VALID] list(..., int maxResult) deprecation is inconsistent | deprecated:
  AbstractQuery.java:6627 list(Class,int) [@Deprecated 6626], 6692 list(RowMapper,int) [6691],
  6847 list(BiRowMapper,int) [6846]; NOT deprecated: 6761 list(RowFilter,RowMapper,int),
  6919 list(BiRowFilter,BiRowMapper,int) | Medium | VERIFIED. The 2-arg maxResult forms are
  @Deprecated ("limit on the DB side") yet the 3-arg maxResult forms they delegate to are not — the
  discouraged concept stays fully reachable without warning.
  Current evidence: `public <T> List<T> list(final Jdbc.RowFilter rowFilter, final Jdbc.RowMapper<? extends T> rowMapper, int maxResult)` (6761) has NO @Deprecated on the preceding lines.
  Recommend: deprecate the (Filter,Mapper,int) variants too, or un-deprecate the 2-arg ones.
- [STILL-VALID] Filter+mapper terminal-op deprecation is inconsistent | deprecated:
  AbstractQuery.java:6244 findFirst(RowFilter,RowMapper) [@Deprecated 6243], 6304

  list(filter,mapper,...), 8738/8848 forEach(filter,...), 8335/8384 count(filter) | Medium | VERIFIED.
  If in-Java row filtering is discouraged enough to deprecate the findFirst filter overloads, the
  analogous list/forEach/count filter overloads are inconsistent in remaining blessed.
  Current evidence: `@deprecated Use {@code stream(RowFilter, RowMapper).first()} instead` (6241) above the deprecated findFirst; `public void forEach(final Jdbc.RowFilter rowFilter, final Jdbc.RowConsumer rowConsumer)` (8738) is NOT deprecated.
  Recommend: pick one policy and apply it across the filter overloads.
