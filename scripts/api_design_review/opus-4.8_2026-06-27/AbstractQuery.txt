1. OVERALL ASSESSMENT
AbstractQuery<Stmt,This> is the fluent base for PreparedQuery/NamedQuery/CallableQuery; the bulk of
it is well-structured and consistent. The setXxx(int, primitive|wrapper|wrapper+default) family is
uniform (primitive, null-handling wrapper, and @Beta default-for-null triplets) with correct JDBC
Types mapping (including the deliberate Float->REAL fix). The terminal-op families are internally
consistent: queryForBoolean/Char/Byte/Short/Int/Long/Float/Double return OptionalXxx;
queryForString/BigInteger/BigDecimal/Date/Time/Timestamp/Bytes return Nullable;
queryForSingleValue/UniqueValue (Nullable) pair cleanly with queryForSingleNonNull/UniqueNonNull
(Optional); findFirst/findOnlyOne (Optional) mirror findFirstOrNull/findOnlyOneOrNull (null); and the
Class / RowMapper / BiRowMapper triad is offered consistently across findFirst, findOnlyOne, list,
stream, insert, batchInsert. The findings concentrate on one genuine duplicate, several
deprecation-consistency gaps, and a few capitalization-based naming hazards.

RE-VERIFICATION SUMMARY: Of the 13 findings, 11 are STILL-VALID, 1 is PARTIALLY-ADDRESSED
(settParameters now carries both a class-doc disambiguation block AND a new `// Rename to
setParametersOnQuery?` TODO comment, but is not renamed), and 1 was already correct and remains so
(NO-LONGER-APPLIES — no action). The known package renames (JdbcUtils -> DataTransferUtil,
ColumnGetter apply->get / get->forType, RowExtractor.createBy->forType, etc.) do NOT touch
AbstractQuery's public surface: this file references none of those renamed symbols
(grep for createBy/createForArray/ColumnGetter/JdbcUtils/getColumnNameList = 0 hits), and its
`.apply(...)` call sites are all RowMapper/BiRowMapper/ResultExtractor (SAM still `apply`).
No NEW public-API inconsistencies were introduced by the recent deletions.

2. NAMING ISSUES
- [STILL-VALID] foreach(Consumer<DisposableObjArray>) / foreach(Class, Consumer<DisposableObjArray>)
  vs the forEach(RowConsumer ...) family | AbstractQuery.java:8806, 8860 (vs forEach 8561/8603/8655/8713) |
  Medium | The two families differ ONLY by letter case (foreach vs forEach) and take different
  callback contracts; a one-character typo silently selects a different method. The Javadoc documents
  the hazard (8773-8776 / 8819-8822) and an in-source "NAMING-REVIEW SUGGESTION (2026-06-27)" comment
  at 8802-8804 / 8856-8858 proposes rename `foreach`, but no rename was made.
  Current evidence: `public void foreach(final Consumer<DisposableObjArray> rowConsumer) throws SQLException { //NOSONAR` (8806);
  `public void foreach(final Class<?> entityClass, final Consumer<DisposableObjArray> rowConsumer) throws SQLException { //NOSONAR` (8860);
  comment `// NAMING-REVIEW SUGGESTION (2026-06-27): \`foreach\` differs from the sibling \`forEach(...)\` only by` (8802).
  Recommend: add an unambiguous name (e.g. foreach) and keep foreach only as a deprecated alias.
- [PARTIALLY-ADDRESSED] settParameters(...) (double-t) vs setParameters(...) | AbstractQuery.java:3153, 3201 |
  Low | Same capitalization/typo hazard; the double-t disambiguates ParametersSetter<? super This>
  from ParametersSetter<? super Stmt>. A class-level doc block (115-125) explicitly explains the
  distinction ("the extra {@code t} is not a typo, and because both compile, the compiler will not
  flag a wrong choice"), and a per-method note exists (3132). The recent commit ADDED an in-source
  TODO `// Rename to setParametersOnQuery?` above each overload (3152, 3200) — so the maintainers
  are now actively considering a rename, yet none has been made; the naming hazard itself remains.
  Note: the new TODO mildly contradicts the doc's "the extra t is not a typo / intentional" framing.
  Current evidence: `public This settParameters(final Jdbc.ParametersSetter<? super This> parametersSetter) throws IllegalArgumentException, SQLException {` (3153);
  `// Rename to setParametersOnQuery?` (3152); class-doc `<li>{@code settParameters(...)} instead hands you <i>this</i> query ({@code This})...` (122).
  Recommend (optional): a clearer name (setParametersOnQuery / configParameters). The confirming note
  from round 1 stands: a single setParameters name would cause lambda-inference ambiguity.

3. RETURN-TYPE AND PARAMETER ISSUES
- [STILL-VALID] query2ResultSets / query3ResultSets accept only BiResultExtractor |
  AbstractQuery.java:5507, 5573 | Low | The single-result query(...) and queryAllResultSets(...) each
  offer BOTH ResultExtractor and BiResultExtractor variants (queryAllResultSets ResultExtractor 5664 /
  BiResultExtractor 5720), but the numbered multi-result-set methods offer only BiResultExtractor.
  (CallableQuery's query2/query3ResultSetsAndGetOutParameters share the limitation — cross-class.)
  Current evidence: `public <R1, R2> Tuple2<R1, R2> query2ResultSets(final Jdbc.BiResultExtractor<? extends R1> resultExtractor1,` (5507);
  `public <R1, R2, R3> Tuple3<R1, R2, R3> query3ResultSets(final Jdbc.BiResultExtractor<? extends R1> resultExtractor1,` (5573).
  Recommend: add the ResultExtractor overloads for parity, or document the intentional restriction.
- [STILL-VALID — intentional omission documented in-source] setParameters(String,String) exists but
  no setParameters(Object,Object) | AbstractQuery.java:2270 (fixed-arity String overloads start at 2
  args) vs 2452 (Object overloads start at 3; the 2-Object form is commented out at 2419-2434) | Low |
  A caller with two Object params cannot use setParameters(a,b) though the String case works. The
  confirming check is ANSWERED by the source: the 2-Object overload is commented out with the note
  "@deprecated to void error: java ambiguous method call with other setParameters methods" (2426) —
  i.e. adding it WOULD create overload-resolution ambiguity. So the omission is deliberate.
  Current evidence: `public This setParameters(final String param1, final String param2) throws SQLException {` (2270);
  `//    public This setParameters(final Object param1, final Object param2) throws SQLException {` (2429, commented).
  Recommend: leave as-is and (optionally) add a one-line note on the String overload pointing callers
  to the 3+-arg Object forms or setParameters(Collection)/setParameters(T, BiParametersSetter) (2806).

6. DEPRECATION / REMOVAL CANDIDATES (with replacement APIs)
- [STILL-VALID] setBigIntegerAsString(int, BigInteger) duplicates setString(int, BigInteger) |
  AbstractQuery.java:1188 vs 1161 | High | Recent commit MOVED setBigIntegerAsString below setString
  but did NOT remove either. Still byte-for-byte equivalent — setBigIntegerAsString's body is literally
  `return setString(parameterIndex, value);` (1189); each @see's the other; both @Beta (1160 / 1187).
  setString(int, BigInteger) binds the BigInteger as a base-10 VARCHAR (1165
  `stmt.setString(parameterIndex, value.toString(10));`).
  Current evidence: `public This setBigIntegerAsString(final int parameterIndex, final BigInteger value) throws SQLException {` (1188), body `return setString(parameterIndex, value);` (1189);
  `public This setString(final int parameterIndex, final BigInteger value) throws SQLException {` (1161).
  Recommend: keep the explicitly-named setBigIntegerAsString and remove the surprising
  setString(int, BigInteger) overload (a setString taking BigInteger is itself misleading).
- [STILL-VALID] list(..., int maxResult) deprecation is inconsistent | deprecated:
  AbstractQuery.java:6629 list(Class,int) [@Deprecated 6628], 6694 list(RowMapper,int), 6849
  list(BiRowMapper,int); NOT deprecated: 6763 list(RowFilter,RowMapper,int),
  6921 list(BiRowFilter,BiRowMapper,int) | Medium | VERIFIED. The 2-arg maxResult forms are
  @Deprecated ("result size should be limited on the database server side ... Use SQL's LIMIT/TOP/ROWNUM")
  yet the 3-arg maxResult forms they delegate to are not — the discouraged concept stays fully
  reachable without warning.
  Current evidence: `@Deprecated` (6628) above `public <T> List<T> list(final Class<? extends T> targetType, final int maxResult) throws SQLException {` (6629);
  `public <T> List<T> list(final Jdbc.RowFilter rowFilter, final Jdbc.RowMapper<? extends T> rowMapper, int maxResult)` (6763) has NO @Deprecated on preceding lines.
  Recommend: deprecate the (Filter,Mapper,int) variants too, or un-deprecate the 2-arg ones.
- [STILL-VALID] Filter+mapper terminal-op deprecation is inconsistent | deprecated:
  AbstractQuery.java:6246 findFirst(RowFilter,RowMapper) [@Deprecated 6245], 6306
  findFirst(BiRowFilter,BiRowMapper), 6431 findFirstOrNull(RowFilter,...), 6512
  findFirstOrNull(BiRowFilter,...) (all "use stream(...).first()"); NOT deprecated: 6763/6921
  list(filter,mapper,...), 8603/8713 forEach(filter,...), 8200/8249 count(filter) | Medium | VERIFIED.
  If in-Java row filtering is discouraged enough to deprecate the findFirst filter overloads, the
  analogous list/forEach/count filter overloads are inconsistent in remaining blessed.
  Current evidence: `@deprecated Use {@code stream(RowFilter, RowMapper).first()} instead` (6243) + `@Deprecated` (6245) above `public <T> Optional<T> findFirst(final Jdbc.RowFilter rowFilter, final Jdbc.RowMapper<? extends T> rowMapper) throws NullPointerException, SQLException {` (6246);
  `public void forEach(final Jdbc.RowFilter rowFilter, final Jdbc.RowConsumer rowConsumer)` (8603) is NOT deprecated.
  Recommend: pick one policy and apply it across the filter overloads.
- [STILL-VALID] count() deprecated while count(RowFilter)/count(BiRowFilter) are not |
  AbstractQuery.java:8158 (deprecated, [@Deprecated 8157]), 8200 (count(RowFilter), @Beta 8199), 8249
  (count(BiRowFilter)) | Low | VERIFIED. count() is deprecated as inefficient (full client-side scan);
  the filtered forms do the same full scan but are kept (justified by predicate logic not expressible
  in SQL). The count(RowFilter) Javadoc acknowledges the cost ("it still requires fetching all rows
  and can be inefficient for large result sets" at 8178-8179), partially addressing the prior note;
  count(BiRowFilter) (8224-) lacks the same explicit cost sentence.
  Current evidence: `@Deprecated` (8157) above `public int count() throws IllegalStateException, SQLException {` (8158);
  `@Beta` (8199) above `public int count(final Jdbc.RowFilter rowFilter) throws IllegalStateException, IllegalArgumentException, SQLException {` (8200).
  Recommend: keep the cost note (already on RowFilter form; consider mirroring on BiRowFilter), or deprecate for consistency.
- [STILL-VALID] @Beta inconsistency within queryForXxx | AbstractQuery.java:4808 queryForBigInteger
  (@Beta 4807), 4842 queryForBigDecimal (@Beta 4841) vs non-Beta queryForString/Date/Time/Timestamp/
  Bytes | Low | These two are @Beta while their sibling Nullable-returning query-for methods are stable.
  Current evidence: `@Beta` (4807) above `public Nullable<BigInteger> queryForBigInteger() throws IllegalStateException, SQLException {` (4808);
  `@Beta` (4841) above `public Nullable<BigDecimal> queryForBigDecimal() throws IllegalStateException, SQLException {` (4842).
  Recommend: promote out of @Beta or document why they remain provisional.
- [STILL-VALID] setLong(int, BigInteger) | AbstractQuery.java:802 | Low | Surprising signature
  (setLong with a BigInteger; throws ArithmeticException on overflow via longValueExact, and closes the
  query before propagating). Mirrored across Named/Callable.
  Current evidence: `public This setLong(final int parameterIndex, final BigInteger value) throws SQLException {` (802),
  body `stmt.setLong(parameterIndex, value.longValueExact());` (809) with `close();` in the finally on failure (813).
  Recommend: keep (cross-family consistency + documentation) or deprecate in all three families together
  toward setBigDecimal/setBigIntegerAsString.
- [NO-LONGER-APPLIES — already correct, no action] setInt(int, char)/setInt(int, Character) |
  AbstractQuery.java:645/670 | Already @Deprecated+@Beta with the correct replacement
  setString(int, char/Character). Remains correct in the current source.
  Current evidence: `@Deprecated` (643) + `@Beta` (644) above `public This setInt(final int parameterIndex, final char value) throws SQLException {` (645);
  `@Deprecated` (668) + `@Beta` (669) above `public This setInt(final int parameterIndex, final Character value) throws SQLException {` (670).
