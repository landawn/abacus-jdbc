API Design Review — JdbcCodeGenerationUtil.java
MODEL=opus-4.8  DATE=2026-06-27  (verified against source by orchestrator)
RE-VERIFIED 2026-06-27 against current source (post Javadoc-fix pass; all line numbers re-located via Grep/Read)

1. OVERALL ASSESSMENT
A healthy, well-organized static codegen utility. Two big families dominate: generateEntityClass (the
DataSource/Connection x table-name/custom-query x with/without EntityCodeConfig matrix) and the
generate{Select,Insert,NamedInsert,Update,NamedUpdate}Sql builders. Both follow a consistent
(DataSource...) -> (Connection...) delegation pattern, return String, throw UncheckedSQLException, and
route through the same checkTableName/checkColumnName/checkColumnLabels validators. The
EntityCodeConfig/JsonXmlConfig Lombok builder pair is well documented. Actionable issues: a real
overload-ambiguity hazard in generateEntityClass (the authors still insert (EntityCodeConfig) null
casts to work around it), a few dead/orphan public members, and convertInsertSqlToUpdateSql breaking the
family's DataSource+Connection convention.

RE-VERIFY SUMMARY: No structural changes since the original report — no generateEntityClassByQuery split
was added, the (EntityCodeConfig) null casts remain, and S/SF/MIN_FUNC/MAX_FUNC/extendFieldNameTableClassName
all still exist. Recent changes were Javadoc-only. Net status: 9/9 findings STILL-VALID (one with a new
nuance: MIN_FUNC/MAX_FUNC now have dedicated unit tests). Line numbers refreshed below.

2. NAMING ISSUES
- STILL-VALID — WON'T FIX (user decision 2026-06-27: no generateEntityClassByQuery split) |
  generateEntityClass(DataSource, String, String) / (Connection, String, String) |
  JdbcCodeGenerationUtil.java:283, 381 | High | The 2nd param is `tableName` in the table-based
  overloads (232/259/337/362) but `entityName` in these query-based overloads, and the 3rd is a raw SQL
  query — the same positional slot carries two meanings across one overload family. No
  generateEntityClassByQuery split was added.
  Evidence (283): `public static String generateEntityClass(final DataSource ds, final String entityName, final String query) {`
  Evidence (381): `public static String generateEntityClass(final Connection conn, final String entityName, final String query) {`
  Recommend: add distinctly named generateEntityClassByQuery(ds/conn, entityName, query[, config]) for the
  query path (this also fixes section 3).

  (DataSource, String, String) [and the Connection pair] | JdbcCodeGenerationUtil.java:259 vs 283; 362 vs 381 | High | These 3-arg
  overloads still differ only in the last param type (EntityCodeConfig vs String).
  generateEntityClass(ds, "users", null) remains ambiguous and does not compile without a cast — the
  internal delegators still write the (EntityCodeConfig) null cast. CONFIRMED both casts present:
  Evidence (233): `return generateEntityClass(ds, tableName, (EntityCodeConfig) null);`
  Evidence (338): `return generateEntityClass(conn, tableName, (EntityCodeConfig) null);`
  Recommend: split the query path into a separately named method (section 2), removing the collision.
- STILL-VALID | generateSelectSql family lacks the excluded-only 3-arg form |
  JdbcCodeGenerationUtil.java:942, 970, 1006, 1037 | Medium | The family still offers only a 2-arg form

  generateNamedInsertSql (1280 DS / 1312 Conn) both provide the excluded-only 3-arg overload. To exclude
  columns from a SELECT a caller must still pass a nullable whereClause.
  Evidence (1006): `public static String generateSelectSql(final DataSource ds, final String tableName, final Collection<String> excludedColumnNames, final String whereClause)`
  Evidence (1143): `public static String generateInsertSql(final DataSource ds, final String tableName, final Collection<String> excludedColumnNames)`

  Evidence (161): `public static final TriFunction<Class<?>, Class<?>, String, String> MAX_FUNC = (entityClass, propClass, propName) -> {`

5. MISSING METHODS (with target class)
- STILL-VALID | convertInsertSqlToUpdateSql(Connection, ...) | JdbcCodeGenerationUtil.java:1859, 1896 |
  Medium | Still exists only as (DataSource, insertSql) and (DataSource, insertSql, whereClause); every
  other generate* family exposes both a DataSource and a Connection overload. A caller holding a Connection
  cannot use it. The method only needs DBProductInfo (resolved via JdbcUtil.getDBProductInfo(ds) at 1898).
  Evidence (1859): `public static String convertInsertSqlToUpdateSql(final DataSource ds, final String insertSql) {`
  Evidence (1896): `public static String convertInsertSqlToUpdateSql(final DataSource ds, final String insertSql, final String whereClause) {`
  Evidence (1898): `final DBProductInfo dbProductInfo = JdbcUtil.getDBProductInfo(ds);`
  Recommend: add convertInsertSqlToUpdateSql(Connection, insertSql[, whereClause]). (A (DBProductInfo, ...)
  or DB-less overload would be even cleaner, but the Connection overload is the minimum to match the family.)

7. NEW ISSUES (introduced/exposed by recent changes)
- None of consequence. The query-path 3-arg delegator generateEntityClass(ds/conn, entityName, query) now
  passes a bare `null` (line 284 / 382) rather than a cast — this is correct because the 4-arg

  (233/338) require the (EntityCodeConfig) null cast. No new API-design defects observed; recent edits were
  Javadoc-only.
