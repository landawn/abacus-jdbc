API Design Review — JdbcCodeGenerationUtil.java
MODEL=opus-4.8  DATE=2026-06-27
RE-VERIFIED (round 2) 2026-06-27 against current committed source (all line numbers re-located via Grep/Read)

1. OVERALL ASSESSMENT
A healthy, well-organized static codegen utility. Two big families dominate: the entity-class generators
and the generate{Select,Insert,NamedInsert,Update,NamedUpdate}Sql builders. Both follow a consistent
(DataSource...) -> (Connection...) delegation pattern, return String, throw UncheckedSQLException, and
route through the same checkTableName/checkColumnName/checkColumnLabels validators.

WHAT CHANGED SINCE LAST REVIEW (the ~116-line committed change): the entity-generator family was SPLIT.
The custom-query path was RENAMED out of generateEntityClass into a new, distinctly named
generateEntityClassByQuery(ds/conn, entityName, query[, config]) family (4 overloads at 330/357/381/409).
The table path (generateEntityClass) now ALWAYS means tableName; the query path
(generateEntityClassByQuery) now ALWAYS means entityName. This is exactly the "split" the previous round
recorded as WON'T-FIX — the user actually performed it. Net effect: the section-2 naming hazard and the
section-3 (EntityCodeConfig vs String) overload collision are BOTH RESOLVED. Separately,
EntityCodeConfig.extendFieldNameTableClassName was removed (commented out), and JsonXmlConfig is now final.

RE-VERIFY SUMMARY (8 original findings):
  ALREADY-FIXED: 2  — (1) generateEntityClass overload ambiguity [naming + return/param collision];

3. RETURN-TYPE AND PARAMETER ISSUES
================================================================================
- ALREADY-FIXED | generateEntityClass 3-arg overload collision (EntityCodeConfig vs String) |
  JdbcCodeGenerationUtil.java:259,310 | The (DataSource/Connection, String, String) query overloads that
  collided with the (.., String, EntityCodeConfig) config overloads were renamed to
  generateEntityClassByQuery. Only one 3-arg generateEntityClass per receiver remains; the call
  generateEntityClass(ds, "users", null) now resolves unambiguously. (See NEW ISSUE N1 re: the leftover
  casts.)
- STILL-VALID | generateSelectSql family lacks the excluded-only 3-arg form |
  JdbcCodeGenerationUtil.java:942,970,1006,1037 | To exclude columns from a SELECT a caller must still pass a
  nullable whereClause; the INSERT and NamedINSERT families both expose the cleaner 3-arg form.
  Recommend: add generateSelectSql(ds/conn, tableName, Collection<String> excludedColumnNames).
- PARTIALLY-FIXED | JsonXmlConfig nested class | JdbcCodeGenerationUtil.java:2403,2443 | Now final
  (consistency defect resolved); still hand-rolls the no-arg ctor (2443) rather than @NoArgsConstructor.
  Recommend: drop the explicit ctor and add @NoArgsConstructor for parity with EntityCodeConfig (2203).

7. NEW ISSUES (introduced/exposed by the generateEntityClassByQuery split)
================================================================================
- N1 (Low, cleanup) | Vestigial (EntityCodeConfig) null casts |
  JdbcCodeGenerationUtil.java:233,286 | The two no-arg delegators still write
  `generateEntityClass(ds/conn, tableName, (EntityCodeConfig) null)`. These casts existed solely to break
  the old (String vs EntityCodeConfig) overload ambiguity; now that the query overloads are renamed away,
  the cast is no longer required to compile and can be reduced to a bare `null` (mirroring the
  generateEntityClassByQuery delegators at 331/382, which already pass bare `null`). Harmless but dead.
  Evidence (233): `return generateEntityClass(ds, tableName, (EntityCodeConfig) null);`
  Evidence (286): `return generateEntityClass(conn, tableName, (EntityCodeConfig) null);`
- No new API-design defects of consequence. The table-path generators correctly delegate through
  generateEntityClassByQuery(conn, tableName, createQueryByTableName(conn, tableName), config) (261/311);
  EntityCodeConfig's @see at 2199 still points at the live generateEntityClass(DataSource,String,
  EntityCodeConfig). Javadoc on the new byQuery overloads is consistent with the family.
