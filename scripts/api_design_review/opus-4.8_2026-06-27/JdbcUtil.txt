1. OVERALL ASSESSMENT
JdbcUtil is a large (~12.1k line) but well-organized static facade and is in healthy shape.
The major families are consistent in naming, ordering, and return types: resource management
(close/closeQuietly with the Connection-closing forms correctly deprecated toward
releaseConnection), the prepareQuery/prepareNamedQuery/prepareCallableQuery factory matrices
({DataSource,Connection} x {autoGeneratedKeys,int[],String[],stmtCreator,ParsedSql}), the
executeQuery/executeUpdate/executeBatchUpdate/executeLargeBatchUpdate/execute execution family,
the extractData / stream / streamAllResultSets / queryByPage extraction families, the
asyncRun/asyncCall arity sets, the blob/clob helpers, and the transaction family
(beginTransaction/runInTransaction/callInTransaction/run|callOutsideTransaction). Findings below
are minor consistency gaps, not structural problems.

  - [REMOVED] getColumnNameList -> getColumnNames : NO LONGER EXISTS. Grep for getColumnNameList

2. NAMING ISSUES
- STILL-VALID | getColumn2FieldNameMap(Class) | JdbcUtil.java:2643 | Medium | Rename to
  getColumn2PropNameMap (or add it and deprecate the old name). The body still delegates directly to
  QueryUtil.getColumn2PropNameMap (2644: `return QueryUtil.getColumn2PropNameMap(entityClass);`), and
  every other entity-metadata accessor here uses the library's "prop" vocabulary
  (getInsertPropNames/getSelectPropNames/getUpdatePropNames, 9470-9630). "FieldName" is the lone
  outlier and contradicts the value it returns. NOTE: a getColumn2PropNameMap facade was NOT added to
  JdbcUtil — the only occurrence is the delegated QueryUtil call.

3. RETURN-TYPE AND PARAMETER ISSUES
- STILL-VALID | skip(ResultSet, long rowsToSkip) | JdbcUtil.java:1740 | Low | Still returns int while
  accepting a long count and accumulating into an int (`int skipped = 0;` at 1747; `skipped++` at
  1752/1761). Skipping > Integer.MAX_VALUE rows overflows the reported total. The sibling
  skip(ResultSet, int) (1706) legitimately returns int. Recommend: change the long overload's return
  type to long. (Confirming check unchanged: a widened return is source-compatible for
  assignment-to-long but not for overload-sensitive callers.)
- STILL-VALID | getInsertPropNames(Object) vs getInsertPropNames(Class) | JdbcUtil.java:9470 / 9515 |
  Low | The Object/Class overload pair still exists and is an easily-confused dispatch: passing a
  Class value selects the class overload, not the instance overload. (getSelectPropNames (9552)/
  getUpdatePropNames (9610) deliberately offer only the Class form.) Recommend: keep, but if confusion
  risk is real, use distinct names (...ForEntity / ...ForClass).
- PARTIALLY-ADDRESSED | getSqlExtractor() / setSqlExtractor(Function) | JdbcUtil.java:10112 / 10133 |
  Low | Asymmetric null contract is UNCHANGED in behavior: setter still stores raw null
  (10134: `JdbcUtil._sqlExtractor = sqlExtractor;`) and getter still returns it raw
  (10113: `return JdbcUtil._sqlExtractor;`); internal logging still defends with
  N.defaultIfNull(JdbcUtil._sqlExtractor, JdbcUtil.DEFAULT_SQL_EXTRACTOR) at 10071. The getter Javadoc
  was UPDATED to document the asymmetry (10110: "... or {@code null} if a {@code null} extractor was
  explicitly set"), so the doc gap is closed but the design recommendation (reject null in setter ->
  fall back to DEFAULT_SQL_EXTRACTOR; guarantee non-null getter) still stands.

4. MISPLACED METHODS (with target class)
- STILL-VALID | isNullOrDefault(Object) | JdbcUtil.java:9819 | Low | Generic value/default check with
  no JDBC dependency (9820-9821: `(value == null) || (value instanceof Number num && num.doubleValue()
  == 0) || (value instanceof Boolean b && !b) || N.equals(value, N.defaultValueOf(value.getClass()))`);
  conceptually belongs in com.landawn.abacus.util.N. Recommend: keep as deliberate DAO/id-layer
  convenience OR move to N.
- STILL-VALID | getInsertPropNames/getSelectPropNames/getUpdatePropNames (9470-9630) and
  getColumn2FieldNameMap (2643) | Low | Pure facade re-exports of QueryUtil; they live correctly in
  QueryUtil and are kept here as facade convenience. No action beyond the section-2 rename.

5. MISSING METHODS (with target class)
- STILL-VALID | prepareNamedQueryForLargeResult(Connection, ParsedSql) | JdbcUtil.java:4285/4319/4352 |
  Medium | The large-result variant exists for (DataSource,String) at 4285, (DataSource,ParsedSql) at
  4319, and (Connection,String) at 4352, but STILL NOT (Connection,ParsedSql) — re-verified: Grep for
  prepareNamedQueryForLargeResult yields exactly those three signatures — even though the non-large
  prepareNamedQuery provides the full {DataSource,Connection}x{String,ParsedSql} matrix. Add it
  (target: JdbcUtil).
- STILL-VALID | public executeLargeUpdate(DataSource|Connection, String, Object...) | JdbcUtil.java |
  Low | executeLargeBatchUpdate is public (DataSource forms 5400/5434, Connection forms 5497/5535) but
  its non-batch counterpart executeLargeUpdate is still only package-private on a PreparedStatement
  (5847: `static long executeLargeUpdate(final PreparedStatement stmt)`). Add a public long-returning
  single-statement variant paralleling executeUpdate (public at 5055/5108).
- STILL-VALID | DataSource overloads for createTableIfNotExists / dropTableIfExists / getColumnNames |
  JdbcUtil.java:8384 / 8433 / 1859 | Low | tableExists offers both (DataSource,...) at 7971 and
  (Connection,...) at 8036; these three related DDL/metadata helpers remain Connection-only
  (createTableIfNotExists(Connection,String,String) 8384; dropTableIfExists(Connection,String) 8433;
  getColumnNames(Connection,String) 1859 — no DataSource overload of any) — an asymmetry. Confirming
  check: whether DDL-via-DataSource is intentionally discouraged.

6. DEPRECATION / REMOVAL CANDIDATES (with replacement APIs)
- UPDATED | Two prior deprecated thin-delegators (getColumnNameList, getColumnLabelList) have ALREADY
  BEEN REMOVED since the original report — confirming the "eligible for outright removal under the
  relaxed backward-compat constraint" path is being exercised. All remaining @Deprecated public methods
  (doesTableExist x2, getColumnValue(label) x2, call/runNotInStartedTransaction x4, close/closeQuietly
  (Connection) x2) still name a correct replacement and are thin delegators; eligible for removal at
  project discretion, not an API defect.

NEW ISSUES (introduced/exposed by recent changes)
- None. The only material change to this file since the original report is the hard-deletion of the two
  deprecated getColumnNameList/getColumnLabelList twins (which removes, rather than introduces, surface).
  The getColumns helpers are now exposed only under the canonical getColumnNames/getColumnLabels names —
  no replacement gap, no orphaned references (Grep for the old names = 0 hits). The getSqlExtractor
  Javadoc was tightened to document the null-return asymmetry (see section 3, PARTIALLY-ADDRESSED). No
  new naming, return-type, placement, or missing-method defects were exposed.
