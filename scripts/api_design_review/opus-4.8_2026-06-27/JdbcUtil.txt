1. OVERALL ASSESSMENT
JdbcUtil is a large (~12.1k line) but well-organized static facade and is in healthy shape. The major
families are consistent in naming, ordering, and return types: resource management (close/closeQuietly with
the Connection-closing forms correctly deprecated toward releaseConnection), the prepareQuery/prepareNamedQuery/
prepareCallableQuery factory matrices ({DataSource,Connection} x {autoGeneratedKeys,int[],String[],stmtCreator,
ParsedSql}), the executeQuery/executeUpdate/executeBatchUpdate/executeLargeBatchUpdate/execute execution family,
the extractData / stream / streamAllResultSets / queryByPage extraction families, the runAsync/callAsync arity
sets, the blob/clob helpers, and the transaction family. Round-2 note: FOUR previously-open findings have been
fixed and committed since round 1 (see ALREADY-FIXED entries). Remaining findings are minor consistency gaps.

- ALREADY-FIXED | skip(ResultSet, long rowsToSkip) | JdbcUtil.java:1740 | (was Low) | Return type widened to
  long and the accumulator is now long — 1740: `public static long skip(final ResultSet rs, long rowsToSkip)`;

  Usage-Example snippet was not updated to the widened type.)
- ALREADY-FIXED | getInsertPropNames(Object) vs getInsertPropNames(Class) "confusable dispatch" |
  JdbcUtil.java:9484/9503 | (was Low) | The Object/Class overload pair is GONE. Only Class forms remain —
  9484: `public static Collection<String> getInsertPropNames(final Class<?> entityClass)` and
  9503: `...getInsertPropNames(final Class<?> entityClass, final Set<String> excludedPropNames)` (-> QueryUtil at
  9504). Now symmetric with getSelectPropNames (9521/9540/9560) and getUpdatePropNames (9579/9598). No Object
  overload exists; the easily-confused dispatch is eliminated.
- ALREADY-FIXED | getSqlExtractor() / setSqlExtractor(Function) null contract | JdbcUtil.java:10081/10103 |
  (was Low/PARTIALLY-ADDRESSED) | Setter now coalesces null to the default —
  10104: `JdbcUtil._sqlExtractor = sqlExtractor == null ? DEFAULT_SQL_EXTRACTOR : sqlExtractor;`. Getter is
  documented and effectively non-null — 10079 `@return ... never {@code null} (defaults to the built-in extractor,

6. DEPRECATION / REMOVAL CANDIDATES (with replacement APIs)
- UNCHANGED | All remaining PUBLIC @Deprecated methods (close/closeQuietly(Connection) x2, getColumnValue(label) x2,
  doesTableExist x2, call/runNotInStartedTransaction x4 — see section 1 for current lines) still name a correct
  replacement and are thin delegators; eligible for outright removal under the relaxed backward-compat constraint
  at project discretion, not an API defect. The getColumnNameList/getColumnLabelList removals (round 1) confirm
  this path is being exercised.

NEW ISSUES (introduced/exposed by recent committed changes)
- NEW | skip(ResultSet, long) Usage-Example snippet not updated to widened return | JdbcUtil.java:1723 | Low (doc) |
  The return type was widened int->long (section 3 ALREADY-FIXED), but the Javadoc example still assigns to an int:
  1723: `int skipped = JdbcUtil.skip(rs, largeNumberOfRowsToSkip);`. Since `skip(ResultSet, long)` now returns
  long, this snippet is a lossy long->int assignment and will NOT compile without a cast. Fix the example to
  `long skipped = JdbcUtil.skip(...)` (and 1725 `"Skipped " + skipped`). This is the only defect exposed by the
  round-2 fixes; the four production fixes (matrix gap, skip widening, sqlExtractor null reset, getInsertPropNames
  Object-overload removal) introduced no new naming, return-type, placement, or missing-method defects.
