API Design Review — JdbcUtil.java
MODEL=opus-4.8  DATE=2026-06-27  (verified against source by orchestrator)

1. OVERALL ASSESSMENT
JdbcUtil is a large (~12k line) but well-organized static facade and is in healthy shape.
The major families are consistent in naming, ordering, and return types: resource management
(close/closeQuietly with the Connection-closing forms correctly deprecated toward
releaseConnection), the prepareQuery/prepareNamedQuery/prepareCallableQuery factory matrices
({DataSource,Connection} x {autoGeneratedKeys,int[],String[],stmtCreator,ParsedSql}), the
executeQuery/executeUpdate/executeBatchUpdate/executeLargeBatchUpdate/execute execution family,
the extractData / stream / streamAllResultSets / queryByPage extraction families, the
asyncRun/asyncCall arity sets, the blob/clob helpers, and the transaction family
(beginTransaction/runInTransaction/callInTransaction/run|callOutsideTransaction). All existing
@Deprecated public methods name a correct replacement and are thin delegators (verified:
getColumnNameList->getColumnNames, getColumnLabelList->getColumnLabels, doesTableExist->tableExists,
label-based getColumnValue perf deprecations, call|runNotInStartedTransaction->*OutsideTransaction,
close|closeQuietly(Connection)->releaseConnection). Findings below are minor consistency gaps,
not structural problems.

2. NAMING ISSUES
- getColumn2FieldNameMap(Class) | JdbcUtil.java:2704 | Medium | Rename to getColumn2PropNameMap
  (or add it and deprecate the old name). The body delegates directly to
  QueryUtil.getColumn2PropNameMap (line 2705), and every other entity-metadata accessor here uses
  the library's "prop" vocabulary (getInsertPropNames/getSelectPropNames/getUpdatePropNames,
  ~9425-9586). "FieldName" is the lone outlier and even contradicts the value it returns.

3. RETURN-TYPE AND PARAMETER ISSUES
- skip(ResultSet, long rowsToSkip) | JdbcUtil.java:1736 | Low | Returns int while accepting a long
  count and accumulating into an int; skipping > Integer.MAX_VALUE rows overflows the reported
  total. The sibling skip(ResultSet, int) (1702) legitimately returns int. Recommend: change the
  long overload's return type to long. Confirming check: confirm callers don't rely on the int
  return (a widened return is source-compatible for assignment-to-long but not for overload-sensitive
  callers).
- getInsertPropNames(Object) vs getInsertPropNames(Class) | JdbcUtil.java:9425 / 9470 | Low | The
  Object/Class overload pair is an easily-confused dispatch: passing a Class value selects the
  class overload, not the instance overload. (getSelectPropNames/getUpdatePropNames deliberately
  offer only the Class form.) Recommend: keep, but if the confusion risk is real, use distinct
  names (...ForEntity / ...ForClass). Confirming check: audit call sites for a Class passed where
  an instance was intended.
- getSqlExtractor() / setSqlExtractor(Function) | JdbcUtil.java:10067 / 10088 | Low | Asymmetric
  null contract: the setter accepts null and the getter then returns null (internal logging defends
  with N.defaultIfNull(...DEFAULT_SQL_EXTRACTOR) at ~10026). Recommend: reject null in the setter
  (fall back to DEFAULT_SQL_EXTRACTOR) and document a guaranteed non-null getter. Confirming check:
  confirm no caller reads back the null it set.

4. MISPLACED METHODS (with target class)
- isNullOrDefault(Object) | JdbcUtil.java:9774 | Low | Generic value/default check with no JDBC
  dependency; conceptually belongs in com.landawn.abacus.util.N. Recommend: keep as a deliberate
  DAO/id-layer convenience OR move to N. Confirming check: whether N already exposes an equivalent.
- getInsertPropNames/getSelectPropNames/getUpdatePropNames (9425-9586) and getColumn2FieldNameMap
  (2704) | Low | Pure facade re-exports of QueryUtil; they live correctly in QueryUtil and are kept
  here only as facade convenience. No action beyond the section-2 rename.

5. MISSING METHODS (with target class)
- prepareNamedQueryForLargeResult(Connection, ParsedSql) | JdbcUtil.java:4346/4380/4413 | Medium |
  The large-result variant exists for (DataSource,String), (DataSource,ParsedSql) and
  (Connection,String) but NOT (Connection,ParsedSql) — verified there is no such overload — even
  though the non-large prepareNamedQuery provides the full {DataSource,Connection}x{String,ParsedSql}
  matrix. Add it (target: JdbcUtil).
- public executeLargeUpdate(DataSource|Connection, String, Object...) | JdbcUtil.java:~5808 | Low |
  executeLargeBatchUpdate is public (5495) but its non-batch counterpart executeLargeUpdate is only
  package-private on PreparedStatement; add a public long-returning single-statement variant paralleling
  executeUpdate (5116/5169). Confirming check: confirm the omission is an oversight, not a deliberate
  "use batch for large" stance.
- DataSource overloads for createTableIfNotExists / dropTableIfExists / getColumnNames |
  JdbcUtil.java:8339 / 8388 / 1855 | Low | tableExists offers both (DataSource,...) and (Connection,...)
  forms; these three related DDL/metadata helpers are Connection-only — an asymmetry. Confirming check:
  whether DDL-via-DataSource is intentionally discouraged.

6. DEPRECATION / REMOVAL CANDIDATES (with replacement APIs)
- None new. All currently @Deprecated public methods already name a correct replacement and are thin
  delegators (verified list in section 1). They are eligible for outright removal under the relaxed
  backward-compat constraint, but that is a project decision, not an API defect.
