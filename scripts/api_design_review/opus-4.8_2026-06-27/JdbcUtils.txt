API Design Review — JdbcUtils.java
MODEL=opus-4.8  DATE=2026-06-27  (verified against source by orchestrator)

1. OVERALL ASSESSMENT
JdbcUtils is a well-structured bulk-data-movement utility (distinct from JdbcUtil) with five healthy
families: importData (Dataset/File/Reader/Iterator -> table), importCsv (File/Reader -> table),
exportCsv (DataSource/Connection/PreparedStatement/ResultSet -> File/Writer), copy
(DataSource/Connection/PreparedStatement, same- or cross-DB), and the createParamSetter factory plus the
new importFrom(...) / DatasetImportBuilder fluent API. Parameter ordering is disciplined
(batchSize/batchIntervalInMillis consistently last before the trailing mapper; selectColumnNames/filter
precede the target), and Csv capitalization is uniform across importCsv/exportCsv. The actionable issues
are (a) one return-type inconsistency in importData, (b) several genuine matrix gaps on the Writer-output
side of exportCsv and the Reader+Connection side of importCsv, and (c) a couple of low-severity parameter
notes largely superseded by the new builder.

2. NAMING ISSUES
- None actionable. importData / importCsv / exportCsv / copy are accurate; Csv capitalization is consistent.
  (importData(File/Reader, ...func) vs importCsv(File/Reader, ...) look overlapping by name but are
  genuinely distinct: the former takes a generic line->Object[] parser, the latter a CSV tokenizer
  producing String[]. Keep both.)

3. RETURN-TYPE AND PARAMETER ISSUES
- importData(Dataset, ...) returns int vs importData(File/Reader/Iterator, ...) returns long |
  JdbcUtils.java:174-989 (all int; builder into(...) 3565/3586/3602 also int) vs 1016, 1122, 1290 (long) |
  Medium | VERIFIED. The same method name returns two different primitive widths. A Dataset is int-bounded
  so int is technically sufficient, but the inconsistent contract is a wart. Recommend: change the Dataset
  importData family AND DatasetImportBuilder.into(...) to return long for family-wide consistency.
- Dataset importData value-mapping arg placement is inconsistent | selectColumnNames at position 2 (239)
  vs columnTypeMap last (353/394) vs stmtSetter last (471/502) | Medium | The three mutually-exclusive
  mapping strategies sit in different positions. This is exactly what importFrom(...) fixes. Recommend:
  keep the builder as the canonical path; do not add further positional permutations.
- batchSize (int) immediately followed by batchIntervalInMillis (long) | e.g. 274, 627, 903 | Low | Two
  adjacent integer-accepting params are silently swappable (long only weakly disambiguates). Inherent to
  the positional design; resolved only by steering callers to the builder's named setters. Recommend: keep.
- DatasetImportBuilder.filter pins the thrown exception to SQLException (3494) vs positional
  importData(..., Throwables.Predicate<? super Object[], E>, ...) (314/440/540) | Low | A filter throwing
  a non-SQLException checked exception works positionally but not via the builder. Recommend: keep
  (acceptable builder simplification). Confirming check: no internal caller needs a non-SQLException
  checked filter.

4. MISPLACED METHODS (with target class)
- None. All bulk import/export/copy methods and createParamSetter (3394) belong in JdbcUtils, not JdbcUtil.

5. MISSING METHODS (with target class)  [all VERIFIED against the exportCsv/importCsv signature matrices]
- exportCsv(PreparedStatement, Writer) and exportCsv(PreparedStatement, Collection<String>, Writer) |
  gap vs File-output equivalents JdbcUtils.java:2082 / 2116 | High | The File side supports PreparedStatement
  (and +cols); the Writer side (2320/2357) only supports ResultSet. You can export a PreparedStatement to a
  File but not to a Writer. Recommend: add both Writer overloads.
- exportCsv(Connection, String querySql, Collection<String> selectColumnNames, Writer) | gap vs 2042
  (Connection+cols->File) and 2277 (Connection->Writer, no cols) | High | Column selection is available for
  File output but not Writer output from a Connection. Recommend: add this overload.
- exportCsv(DataSource, String querySql, Collection<String> selectColumnNames, File/Writer) | gap vs 1963 /
  2234 (DataSource variants have no selectColumnNames) | Medium | The DataSource entry points cannot select
  columns at all. Recommend: add the two column-selecting DataSource overloads (or document DataSource as
  all-columns-only).
- importCsv(Reader, Connection, String insertSql, int batchSize, long batchIntervalInMillis, stmtSetter) |
  gap vs 1540 (importCsv(File, Connection, ...) exists) and 1165 (importData(Reader, Connection, ...) exists)
  | High | The Reader CSV family (1717/1761/1804) has NO Connection overload though both its File sibling and
  the parallel importData(Reader) family do. Recommend: add the Reader+Connection importCsv overload.
- Configured Dataset import via DataSource | only the 3-arg importData(Dataset, DataSource, String) exists
  (174) | Low | Every configured Dataset import via a DataSource currently requires a Connection or the
  builder. Recommend: KEEP — do not add positional DataSource overloads; importFrom(dataset)...into(dataSource,
  sql) (3565) already covers this ergonomically.

6. DEPRECATION / REMOVAL CANDIDATES (with replacement APIs)
- No @Deprecated members currently exist. Optional/Low: if the team wants to curb the ~30-overload
  positional importData(Dataset, ...) surface, the natural replacement is importFrom(Dataset) +
  DatasetImportBuilder (3437/3459). Do NOT mass-deprecate now — the positional path is the documented,
  tested one and the builder is brand-new/unmarked; revisit after bake-in. No removals recommended.
