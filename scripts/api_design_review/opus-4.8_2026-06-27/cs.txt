1. OVERALL ASSESSMENT
cs is a utility holder of ~95 public static final String constants whose values are the literal
parameter names used across the library (consumed as N.checkArgNotNull(x, cs.x)). The convention is
camelCase identifiers whose value equals the identifier. The set is internally consistent; the only
actionable item is a single casing/duplication outlier, which is still present.

2. NAMING ISSUES
- [STILL-VALID] ResultSet ("ResultSet", capitalized) coexisting with resultSet ("resultSet") |
  cs.java:56 and 252 | Low | Current evidence:
  cs.java:56  -> "public static final String ResultSet = "ResultSet";"
  cs.java:252 -> "public static final String resultSet = "resultSet"; // NOSONAR"
  Every other constant is camelCase with a lowercase first letter; ResultSet is the lone capitalized
  identifier, and a near-duplicate resultSet already exists with a different value. Two constants
  differing only by first-letter case and value remain confusing. Recommend: standardize on resultSet and
  update the few call sites whose parameter is literally named "ResultSet" to match, then remove the
  capitalized constant. Confirming check: grep cs.ResultSet usages and confirm the underlying parameter
  name can be lower-cased without breaking a reflective/name-based lookup.
