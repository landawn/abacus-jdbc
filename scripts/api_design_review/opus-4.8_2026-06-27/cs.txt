API Design Review — cs.java (public final class of parameter-name String constants)
MODEL=opus-4.8  DATE=2026-06-27  (reviewed directly by orchestrator)

1. OVERALL ASSESSMENT
cs is a utility holder of ~90 public static final String constants whose values are the literal
parameter names used across the library (consumed as N.checkArgNotNull(x, cs.x)). The convention is
camelCase identifiers whose value equals the identifier. The set is internally consistent; the only
actionable item is a single casing/duplication outlier.

2. NAMING ISSUES
- ResultSet ("ResultSet", capitalized) coexisting with resultSet ("resultSet") | cs.java:56 and 252 | Low |
  Every other constant is camelCase with a lowercase first letter; ResultSet is the lone capitalized
  identifier, and a near-duplicate resultSet already exists with a different value. This is confusing (two
  constants differing only by first-letter case and value). Recommend: standardize on resultSet and update
  the few call sites whose parameter is literally named "ResultSet" to match, then remove the capitalized
  constant. Confirming check: grep for cs.ResultSet usages and confirm the underlying parameter name can be
  lower-cased without breaking a reflective/name-based lookup.

3. RETURN-TYPE AND PARAMETER ISSUES
- None (constants only). Note: the cluster targetClass / targetType / targetValueType / cls / entityClass
  (cs.java:328/332/356/424/144) mirrors an underlying parameter-naming inconsistency in the real APIs; that
  is a finding for those classes, not for cs (which faithfully reflects the actual names).

4. MISPLACED METHODS — None.
5. MISSING METHODS — None (constants are added as new parameter names appear).
6. DEPRECATION / REMOVAL CANDIDATES — None beyond folding ResultSet into resultSet (section 2).
