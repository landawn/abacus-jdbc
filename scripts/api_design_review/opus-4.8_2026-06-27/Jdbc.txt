1. OVERALL ASSESSMENT
Unchanged from round 1: Jdbc is a mature namespace of well-aligned Bi-mirror functional-interface families.
Round-1 fixes closed 3 findings (HandlerFactory arity-ambiguity, OutParamResult mutable-view leak, DaoCache
createByMap(int) gap) and made 1 finding moot (OutParam invariant — now documented sentinel). The remaining open
findings are mirror gaps, two cross-family naming nits, and the still-invariant ColumnOne.set generic. The for*-verb
migration is intact (ColumnGetter.forType, RowExtractor.forType, forArray/forList, forEachColumn); HandlerFactory's
new createForBeforeInvoke/createForAfterInvoke keep a create* prefix rather than a bare for* verb (minor — see §7).

2. NAMING ISSUES
- [ALREADY-FIXED] HandlerFactory.create single-arg arity-ambiguity | Jdbc.java:6748,6780,6817 | (was Medium) | The two
  single-arg overloads were renamed: 6748 "public static <T> Handler<T> createForBeforeInvoke(" and 6780
  "public static <T> Handler<T> createForAfterInvoke(". The 2-arg create(before, after) was kept at 6817
  "public static <T, E extends RuntimeException> Handler<T> create(". Javadoc examples updated to the new names
  (6739 createForBeforeInvoke((Throwables.TriConsumer...)), 6771 createForAfterInvoke((Throwables.QuadConsumer...))).
  Intent is now explicit; arity-disambiguation no longer required. Closed.
- [STILL-VALID] RowConsumer#oneOff / BiRowConsumer#oneOff vs RowMapper#toDisposableObjArray /
  BiRowMapper#toDisposableObjArray | Jdbc.java:4587,4629,4790,4832 (oneOff) vs 2022,2070,3863,3904
  (toDisposableObjArray) | Medium-Low | Same "wrap each row into a reusable DisposableObjArray" concept named
  toDisposableObjArray on mappers but oneOff on consumers. 4587: "static RowConsumer oneOff(final
  Consumer<DisposableObjArray> consumer) {"  vs  2022: "static RowMapper<DisposableObjArray> toDisposableObjArray() {".
  Still sharper because the sibling per-column factory is the descriptive forEachColumn (4541, 4762) sitting beside
  the opaque oneOff inside the SAME interface. Recommend rename to toDisposableObjArray/consumeDisposableObjArray or document.
- [STILL-VALID] DaoCache#create(int,long) vs #createByMap(...) | Jdbc.java:6870, 6880, 6892, 6905 | Low | create is the
  generic name yet denotes the pooled/TTL-evicting variant: 6870 "static DaoCache create(final int capacity, final long
  evictDelay) {" while the map-backed family is the descriptive createByMap (6880 createByMap(), 6892 createByMap(Map),
  6905 createByMap(int) — the last newly added). Recommend createWithEviction/createPooled for the first so the pair
  self-describes. (createByMap intentionally not renamed.) Pure-rename; BC relaxed.

3. RETURN-TYPE AND PARAMETER ISSUES
- [STILL-VALID] Columns.ColumnOne#set(Class<T>) | Jdbc.java:6179 | Low | Still invariant Class<T> whereas every sibling
  type-keyed factory uses Class<? extends T>: 6179 "public static <T> BiParametersSetter<AbstractQuery, T> set(final
  Class<T> type) {" vs ColumnOne.get 6091 "get(final Class<? extends T> firstColumnType)", ColumnGetter.forType 5752
  "forType(final Class<? extends T> cls)". NOTE: change-list claimed this was widened — NOT reflected in committed source.
  Recommend set(final Class<? extends T> type).
- [ALREADY-FIXED] OutParamResult#getOutParamValues() / #getOutParams() | Jdbc.java:6384, 6401 | (was Low) | Now return
  immutable wrappers: 6385 "return ImmutableMap.wrap(outParamValues);" and 6402 "return ImmutableList.wrap(outParams);".
  Javadoc updated to "an unmodifiable view ..." (6382, 6399). External mutation no longer possible. Closed.
- [NO-LONGER-APPLIES] OutParam parameterIndex>0 not enforced in ctor/setter | Jdbc.java:6238 (class), 6285 (of) | Per
  do-not-reflag directive: index<=0 is the LIVE SENTINEL for name-based OUT-param registration (CallableQuery byIndex =
  getParameterIndex() > 0; JdbcUtil if(outParam.getParameterIndex() > 0)). of(int,int) correctly validates the index-based
  path: 6286 "N.checkArgument(parameterIndex > 0, ...)". Class-doc now documents this intentionally (6213-6216: "accept
  any value (e.g. a placeholder 0 to be set later)"). Not a defect.

4. MISPLACED METHODS
- None material (unchanged). ColumnOne.set(...) returning a BiParametersSetter writer under a "read column 1" type is the
  deliberate, documented "first parameter" symmetry.

5. MISSING METHODS
- [STILL-VALID] BiResultExtractor lacks the Dataset / merged-list mirror of ResultExtractor | BiResultExtractor interface
  at 1223 (TO_DATASET constant 1231, toList 1738) — but NO toMergedList/toDataset/toDatasetAndThen factories | Medium |
  ResultExtractor has toMergedList (1015/1043/1073), toDataset (1097/1120/1141/1152/1172), toDatasetAndThen (1192); the Bi
  mirror has only the TO_DATASET constant (1231 "BiResultExtractor<Dataset> TO_DATASET = (rs, columnLabels) -> {") and
  toList(Class) (1738). These produce label-aware Datasets and ignore the columnLabels arg, so each could be a thin
  delegate. Recommend add the mirrors or document the asymmetry.
- [ALREADY-FIXED] DaoCache#createByMap(int capacity) | Jdbc.java:6905 | (was Low) | The capacity-sized map-backed
  constructor is now reachable: 6905 "static DaoCache createByMap(final int capacity) {" -> 6906 "return new
  DaoCacheByMap(capacity);" (ctor at 7196 "public DaoCacheByMap(final int capacity) {"). @see entries updated
  (7180-7181). Closed. (Residual minor: DaoCache class-doc still @see/@link the package-private DaoCacheByMap at
  6854/6858 and 7180-7181 — non-public doc exposure; unchanged, Low.)

7. NEW ISSUES / RE-VERIFICATION NOTES (round 2)
- [NEW — Low, naming consistency] HandlerFactory's new factories use a create* prefix, not a bare for* verb. The round-1
  rename produced createForBeforeInvoke (6748) / createForAfterInvoke (6780), whereas the rest of the for*-migration uses
  bare verbs (ColumnGetter.forType 5752/5770, RowExtractor.forType 5160..5216, BiParametersSetter.forArray 274/forList 342,
  RowConsumer/BiRowConsumer.forEachColumn 4541/4762). Minor: these produce a Handler so a create* prefix is defensible;
  flag only as a consistency observation, not a defect.
- [UPDATED — §2/§7 "create-named outliers" now reduced] After the HandlerFactory single-arg rename, the only remaining
  bare create-named factories are HandlerFactory.create(before,after) (6817, intentionally kept) and DaoCache.create(int,long)
  (6870) + the createByMap family (6880/6892/6905). The mid-migration visual inconsistency is smaller than round 1; the
  surviving DaoCache.create naming nit (§2) stands.
- [STILL-VALID — Low, amplifies §2] forEachColumn vs oneOff style clash WITHIN RowConsumer/BiRowConsumer. Descriptive
  forEachColumn (4541, 4762) sits beside opaque oneOff (4587/4629 and 4790/4832) in each consumer interface.
- [RESOLVED — do NOT re-flag] ColumnGetter and RowExtractor type-keyed factories share the forType verb: 5752 "static <T>
  ColumnGetter<T> forType(final Class<? extends T> cls)" and 5160 "static RowExtractor forType(final Class<?>
  entityClassForFetch)". Prior cross-family nit resolved.
- [INTENTIONAL — do NOT re-flag] ColumnOne keeps get/set/readJson/readXml while ColumnGetter moved get->forType.
  ColumnOne.get(Class/Type) [6091/6113] returns a RowMapper reading COLUMN 1, a different concept from ColumnGetter.forType
  building a reusable getter. Divergent verbs are deliberate; not a regression.
