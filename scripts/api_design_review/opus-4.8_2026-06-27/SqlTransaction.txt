API Design Review — SqlTransaction.java
MODEL=opus-4.8  DATE=2026-06-27  (verified against source by orchestrator)

1. OVERALL ASSESSMENT
SqlTransaction is a final class implementing Transaction + AutoCloseable (line 45) and is handed out as
the concrete type by JdbcUtil.beginTransaction(...), so its extra public methods beyond the interface
(connection(), runOutsideTransaction, callOutsideTransaction) are reachable by normal callers without
casting. The accessor family (id() 166, isolationLevel() 216, status() 238, isActive() 262) is healthy,
accurate, and consistent with the interface and with the class's fluent no-"get" convention. The lifecycle
set (commit() 306, rollback() 431, rollbackIfNotCommitted() 500, close() 1033) is correctly implemented
with sound double-commit / close-after-commit / nested-scope semantics, delegating to package-private
commit(Runnable)/rollback(Runnable) workhorses. The only actionable issues are deprecation hygiene.

2. NAMING ISSUES
- runNotInMe(Throwables.Runnable) / callNotInMe(Throwables.Callable) | SqlTransaction.java:963, 1012 | High |
  The "NotInMe" names are non-descriptive and inconsistent with the rest of the surface; already superseded
  by the clearer runOutsideTransaction/callOutsideTransaction (833/899). Treated as one unit — see section 6
  (remove). connection() (194) intentionally follows the no-"get" accessor style (matching id/status/
  isolationLevel) and is consistent, not a defect.

3. RETURN-TYPE AND PARAMETER ISSUES
- None actionable. The void-vs-R split across runOutsideTransaction/callOutsideTransaction (and the
  deprecated runNotInMe/callNotInMe) is the correct Runnable-vs-Callable mirror. The post-action overloads
  carrying a Runnable are package-private (out of the public surface).

4. MISPLACED METHODS
- None. connection() (194), runOutsideTransaction (833) and callOutsideTransaction (899) operate on this
  transaction's owned connection / thread registration and belong here.

5. MISSING METHODS (with target class)
- connection(), runOutsideTransaction(...), callOutsideTransaction(...) exist only on the concrete
  SqlTransaction, not on the Transaction interface | SqlTransaction.java:194, 833, 899 | Low | Acceptable
  because JdbcUtil.beginTransaction returns SqlTransaction (not Transaction), so callers get them. Recommend:
  keep as-is unless the library starts returning the Transaction interface type. Confirming check: grep for
  any public API returning the bare Transaction type (none found).

6. DEPRECATION / REMOVAL CANDIDATES (with replacement APIs)
- runNotInMe / callNotInMe | SqlTransaction.java:963, 1012 | High | VERIFIED both @Deprecated, pure one-line
  delegations to runOutsideTransaction / callOutsideTransaction. Recommend: remove (replacement =
  *OutsideTransaction). BC relaxed.
- rollback() | SqlTransaction.java:431 | Medium | VERIFIED @Deprecated in favor of rollbackIfNotCommitted()
  (500). However the deprecation is NOT reflected on the Transaction interface — Transaction.rollback()
  (Transaction.java:165) is not @Deprecated (only prose at 158-159 mentions it, and the Action-enum Javadoc
  example at Transaction.java:266-271 still calls transaction.rollback()). Recommend: for cross-type
  consistency, also @Deprecated Transaction.rollback() and update its example to rollbackIfNotCommitted().
  (See SUMMARY and the Transaction report.)
- CreatedBy.SQL_EXECUTOR | SqlTransaction.java:~1113 | Low | Already @Deprecated ("no longer used"), and the
  enclosing CreatedBy enum is package-private (out of the public surface). Keep for ordinal stability; no
  public-API action.
