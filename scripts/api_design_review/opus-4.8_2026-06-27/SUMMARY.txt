MODEL=opus-4.8   DATE=2026-06-27
Mode: PARALLEL (one sub-agent per large class to surface candidates; orchestrator verified EVERY finding

Scope: 22 public top-level types. Package-private files (DaoImpl, JdbcSettings, ResultSetProxy, SqlExecutor,
      SqlLogConfig) excluded — no public API surface.

Overall: the package is mature and largely consistent. The fluent query hierarchy, the JdbcUtil facade
families, the Jdbc functional-interface mirrors, and the JdbcUtils bulk-IO families are all in good shape,
and existing @Deprecated members nearly always name a correct replacement. The findings are consistency and
hygiene issues, not structural defects. One genuine duplicate (setBigIntegerAsString == setString(BigInteger))
is the only High-confidence "two methods do exactly the same thing" case.

============================================================
1. CONSISTENCY ISSUES SPANNING MULTIPLE CLASSES
============================================================

A. Case-only / arity-only overload disambiguation (lambda-ambiguity workarounds) — AbstractQuery, Jdbc.
   foreach vs forEach and settParameters vs setParameters (AbstractQuery 8925/8975, 3140/3187), and
   HandlerFactory.create's two single-arg overloads distinguished only by lambda arity (Jdbc 6614/6646).
   These are intent-unsafe: a typo or a casual reader silently selects the wrong method. Want one uniform,
   explicit disambiguation-naming convention (e.g. forEachRow / configParameters / createForBeforeInvoke /
   createForAfterInvoke).

B. Named-setter mirror divergence: NamedQuery vs CallableQuery (mirror by design).
   CallableQuery lacks setObject(String,Object,SQLType[,int]), setParameters(Object) bean, and
   setNString(String,CharSequence); uses List<String> where NamedQuery uses Collection<String>; and
   NamedQuery.setNull(String,int)/setString(String,String) carry a narrower throws clause than the rest of
   the family. Align the two named families (and their throws clauses).

C. Numbered multi-result-set methods accept only BiResultExtractor while single query() accepts both —
   in BOTH AbstractQuery (query2/query3ResultSets) and CallableQuery (...AndGetOutParameters). Add the
   ResultExtractor overloads or document the restriction uniformly.

D. Deprecation-policy inconsistency inside AbstractQuery: a deprecated method's replacement is still reachable
   un-deprecated (list maxResult 2-arg vs 3-arg; findFirst(filter) deprecated but list/forEach/count(filter)
   not; count() deprecated but count(filter) not). Pick one stance for in-Java filtering and for maxResult.

E. Incomplete factory matrices (DataSource/Connection and output-target gaps) — JdbcUtil
   (prepareNamedQueryForLargeResult missing (Connection,ParsedSql); DDL helpers Connection-only),
   JdbcCodeGenerationUtil (convertInsertSqlToUpdateSql DataSource-only), JdbcUtils (exportCsv Writer side and
   importCsv Reader side both miss overloads their File/Connection siblings have).

F. Row-count width int vs long — JdbcUtils.importData (int for Dataset, long for File/Reader/Iterator) and
   JdbcUtil.skip(long)->int. Standardize on long for "rows affected/processed".

G. Enum int-accessor convention — intValue()/valueOf(int) is the convention (IsolationLevel, FetchDirection);
   OnDeleteAction diverges with value()/get(String) but is already @Deprecated (no action unless revived).

H. null vs Optional/Nullable returns — DBLock.lock() and SpringApplicationContext.getBean return null where the
   library otherwise favors Optional/Nullable. Low priority (both have idiomatic justifications).

I. Cross-type deprecation gap — SqlTransaction.rollback() is @Deprecated but the Transaction interface's
   rollback() is not (and its Action example still calls it).

J. Dead/orphan public members & removable deprecated aliases — DBLock.UNLOCKED; JdbcCodeGenerationUtil
   S/SF/MIN_FUNC/MAX_FUNC/extendFieldNameTableClassName; JoinInfo.setNullSqlAndParamSetterPool (built+unused)
   and its 4 deprecated *SqlBuilderAndParamSetter aliases; SqlTransaction runNotInMe/callNotInMe.

K. Vocabulary drift — JdbcUtil.getColumn2FieldNameMap ("Field") vs the "Prop" naming everywhere else;
   targetType/targetValueType/targetClass/cls param-name drift (visible in cs.java).
