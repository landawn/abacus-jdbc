API Design Review — NamedQuery.java
MODEL=opus-4.8  DATE=2026-06-27  RE-VERIFIED (round 2) 2026-06-27 against current committed source
(every finding re-checked against CURRENT NamedQuery.java + CallableQuery.java + AbstractQuery.java; line numbers refreshed; STATUS-prefixed)

1. OVERALL ASSESSMENT
NamedQuery extends AbstractQuery<PreparedStatement, NamedQuery> and inherits the entire positional
setXxx(int,...) + terminal-op API; its own contribution is the named-parameter setter family
setXxx(String,...), which is broad and overwhelmingly consistent: every setter returns NamedQuery for
chaining, every method puts the parameter name first, boxed-primitive overloads uniformly translate
null into setNull(name, <correct Types>) (including the Float->REAL fix), and the multi-occurrence
binding ladder is identical across all setters. Coverage is richer than CallableQuery's mirror (NamedQuery
adds setNString(CharSequence) 1329, setRef 3278, setArray 3343, setObject(...SQLType) 3637/3711,
setObject(...Type<T>) 3786). Unlike CallableQuery, NamedQuery resolves each name to its 1-based index(es)
itself and throws IllegalArgumentException("Named parameter not found") on an unknown name — so its
setters legitimately declare `throws IllegalArgumentException, SQLException`. ROUND-2 KEY CHANGE: the prior
throws-clause inconsistency (setNull(String,int) and setString(String,String) declaring SQLException only)
is now ALREADY-FIXED — both declare the full `throws IllegalArgumentException, SQLException`. Remaining
findings are one documented strict-vs-lenient asymmetry and a couple of low-impact parity gaps vs the
positional family.

5. MISSING METHODS (with target class)
- STATUS: STILL-VALID (intentional omission, parity choice) — Boxed-primitive "default-for-null" overloads
  absent from the named family | NamedQuery.java setBoolean/setByte/setShort/setInt/setLong/setFloat/
  setDouble | Low | The positional family in AbstractQuery provides setXxx(int, Boxed, primitiveDefault)
  (AbstractQuery.java:380 setBoolean, 449 setByte, 518 setShort, 589 setInt, 741 setLong, 885 setFloat,
  983 setDouble — all re-confirmed this pass); the named family still provides no (String, Boxed,
  primitiveDefault) overload (grep for `defaultValueForNull` in NamedQuery = 0 matches). CallableQuery's
  named family also omits these -> consistent, likely an intentional "keep the named API leaner" choice.
  Recommend: add for parity OR accept deliberately.
- STATUS: STILL-VALID (intentional omission, parity choice) — Calendar-aware date/time overloads absent |
  NamedQuery.java setDate/setTime/setTimestamp | Low | AbstractQuery offers setDate(int, java.sql.Date,
  Calendar) (1287), setTime(int, Time, Calendar) (1369), setTimestamp(int, Timestamp, Calendar) (1451 — all
  re-confirmed); the named family has no (String,...,Calendar) equivalent. CallableQuery also omits them ->
  consistent, likely intentional. Recommend: add for parity or accept. (Healthy, not a gap: the named family
  deliberately exposes only the long-length setAsciiStream/setBinaryStream/setCharacterStream overloads,
  matching CallableQuery.)

  setInt(String, Character) | NamedQuery.java:660 (@Deprecated 659) | Both carry @Deprecated with
  replacement setString(String, char/Character); mirror the positional/Callable families. No change.
- STATUS: STILL-VALID (keep) — setLong(String, BigInteger) | NamedQuery.java:784 | Low | Still converts
  via `value.longValueExact()` (790) and can throw ArithmeticException for out-of-range values — surprising

  AbstractQuery/CallableQuery.)
