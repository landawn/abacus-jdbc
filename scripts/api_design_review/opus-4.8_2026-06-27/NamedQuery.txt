API Design Review — NamedQuery.java
MODEL=opus-4.8  DATE=2026-06-27  RE-VERIFIED 2026-06-27 against current source
(every finding re-checked against CURRENT NamedQuery.java + CallableQuery.java + AbstractQuery.java; line numbers refreshed)

1. OVERALL ASSESSMENT
NamedQuery extends AbstractQuery<PreparedStatement, NamedQuery> and inherits the entire positional
setXxx(int,...) + terminal-op API; its own contribution is the named-parameter setter family
setXxx(String,...), which is broad and overwhelmingly consistent: every setter returns NamedQuery for
chaining, every method puts the parameter name first, boxed-primitive overloads uniformly translate
null into setNull(name, <correct Types>) (including the Float->REAL fix), and the multi-occurrence
binding ladder is identical across all setters. Coverage is richer than CallableQuery's mirror (NamedQuery
adds setNString(CharSequence) 1329, setRef 3278, setArray 3343, setObject(...SQLType) 3637/3711,
setObject(...Type<T>) 3786). Unlike CallableQuery, NamedQuery resolves each name to its 1-based index(es)
itself and throws IllegalArgumentException("Named parameter not found") on an unknown name — so its
setters legitimately declare `throws IllegalArgumentException, SQLException`. Findings are limited to a
small throws-clause inconsistency (two intra-class outliers), one documented strict-vs-lenient asymmetry,
and a couple of low-impact parity gaps vs the positional family.

3. RETURN-TYPE AND PARAMETER ISSUES
- STATUS: STILL-VALID — Throws-clause inconsistency: setNull(String,int) | NamedQuery.java:153 and
  setString(String,String) | NamedQuery.java:1112 | Medium | Both still declare only `throws SQLException`,
  while every other setter in the SAME class declares `throws IllegalArgumentException, SQLException` AND
  both outliers in fact throw IAE and already @throws-document it. Evidence (current source):
    153 : `public NamedQuery setNull(final String parameterName, final int sqlType) throws SQLException {`
          ...body 166/177: `throw new IllegalArgumentException("Named parameter not found: " + parameterName);`
          ...javadoc 149: `@throws IllegalArgumentException if the parameter name is not found in the SQL query`
    1112: `public NamedQuery setString(final String parameterName, final String value) throws SQLException {`
          ...javadoc 1109: `@throws IllegalArgumentException if the parameter name is not found in the SQL query`
   Family reference: setNull(String,int,String):218 and setString(String,CharSequence):1176 both declare
   `throws IllegalArgumentException, SQLException`. Recommend: add IllegalArgumentException to the declared
   throws of lines 153 and 1112 so signature matches the rest of the family and their own Javadoc.
   CROSS-TYPE NOTE (REFINED this pass — the old "pick one direction" advice was misleading): do NOT
   propagate to CallableQuery. CallableQuery's named setters delegate names straight to the driver
   (no name->index lookup, no IAE), so their `throws SQLException`-only clauses are correct there. The fix
   belongs ONLY in NamedQuery's two outliers.
- STATUS: STILL-VALID / PARTIALLY-ADDRESSED — setParameters(Map) lenient vs setParameters(Object bean) /
  setParameters(Object, Collection) strict | NamedQuery.java:3858 (Map, lenient) vs 3932 (bean, strict) /
  4025 (bean+names, strict) | Low | Behavior unchanged: the Map overload binds only names present in the
  map and leaves missing named params unbound (body 3862-3866), while the bean overloads throw
  IAE("No property found with name: ...") when a named param has no matching property (3945-3950 /
  4045-4047). Same family, opposite absence handling. PARTIALLY-ADDRESSED: each Javadoc now documents its
  OWN behavior (Map 3839-3841 "...left unbound"; bean 3926-3927 "@throws IAE ... lacks a property") and
  CallableQuery's Map javadoc (CallableQuery.java:1525-1527) now explicitly contrasts the two, but
  NamedQuery's own Javadocs do not cross-reference each other. Recommend: add a one-line cross-reference
  in NamedQuery's Map/bean Javadocs (or accept as intentional product behavior).

5. MISSING METHODS (with target class)
- STATUS: STILL-VALID (intentional omission, parity choice) — Boxed-primitive "default-for-null" overloads
  absent from the named family | NamedQuery.java setBoolean/setByte/setShort/setInt/setLong/setFloat/
  setDouble | Low | The positional family in AbstractQuery provides setXxx(int, Boxed, primitiveDefault)
  (AbstractQuery.java:380 setBoolean, 449 setByte, 518 setShort, 589 setInt, 741 setLong, 885 setFloat,
  983 setDouble); the named family still provides no (String, Boxed, primitiveDefault) overload (grep
  confirms zero matches). CallableQuery's named family also omits these -> consistent, likely an
  intentional "keep the named API leaner" choice. Recommend: add for parity OR accept deliberately.
- STATUS: STILL-VALID (intentional omission, parity choice) — Calendar-aware date/time overloads absent |
  NamedQuery.java setDate/setTime/setTimestamp | Low | AbstractQuery offers setDate(int, java.sql.Date,
  Calendar) (1287), setTime(int, Time, Calendar) (1369), setTimestamp(int, Timestamp, Calendar) (1451);
  the named family has no (String,...,Calendar) equivalent (the only "Calendar" tokens in NamedQuery are
  Javadoc examples at 1518/1651, not signatures). CallableQuery also omits them -> consistent, likely
  intentional. Recommend: add for parity or accept. (Healthy, not a gap: the named family deliberately
  exposes only the long-length setAsciiStream/setBinaryStream/setCharacterStream overloads, matching
  CallableQuery.)

6. DEPRECATION / REMOVAL CANDIDATES (with replacement APIs)
- STATUS: STILL-VALID (already handled) — setInt(String, char) | NamedQuery.java:635 and
  setInt(String, Character) | NamedQuery.java:660 | Both still carry @Deprecated (634, 659) with
  replacement setString(String, char/Character); mirror the positional/Callable families. No change.
- STATUS: STILL-VALID (keep) — setLong(String, BigInteger) | NamedQuery.java:784 | Low | Still converts
  via `value.longValueExact()` (790) and can throw ArithmeticException for out-of-range values — surprising
  for a setLong, but documented (`@throws ArithmeticException` 781) and consistent across all three
  families; null -> setNull(...,BIGINT) (786). Recommend: keep, or (if intent is to discourage) deprecate
  in all three families simultaneously toward setBigDecimal/setBigIntegerAsString. (Cross-class with
  AbstractQuery/CallableQuery.)
